# Korean Text Safety

이 문서는 PathClash 프로젝트에서 한글 문자열을 수정하거나 추가할 때 지켜야 하는 안전 규칙을 정리한 문서입니다.
목적은 인코딩 오판, 불필요한 문자열 변경, `???` 같은 잘못된 치환을 막는 것입니다.

## 기본 원칙

- 기존에 정상적으로 보이는 한글 문자열은 함부로 바꾸지 않습니다.
- 한글 수정은 요청 범위 안에서만 최소 범위로 진행합니다.
- 실제 사용자에게 보이는 UI 기준으로 판단합니다.
- 터미널 출력은 참고용일 뿐, 소스 원본 판단 기준이 아닙니다.

## 한글/인코딩 안전 규칙

- 터미널이나 PowerShell 출력에서 한글이 깨져 보여도, 그것만 근거로 기존 한글 문자열을 수정하지 않습니다.
- 실제 원본 판단은 브라우저 UI, 에디터 표시, git diff의 기존값을 우선합니다.
- 요청 범위 밖의 기존 한글 문구는 인코딩이 의심되더라도 함부로 바꾸지 않습니다.
- 새 한글 문구를 추가할 때도 기존 정상 한글을 다른 형태로 치환하지 않습니다.
- 파일 맨 앞에 BOM이 생기지 않도록 주의합니다.

## 작업 절차

1. 먼저 실제 브라우저/UI에서 한글이 어떻게 보이는지 확인합니다.
2. 수정 범위를 파일과 줄 또는 블록 단위로 최대한 좁힙니다.
3. 사용자가 직접 준 한글 문구가 있으면 그대로 사용합니다.
4. 기존 한글과 무관한 리팩터링, 대규모 포매팅, 문자열 정리는 함께 하지 않습니다.
5. 수정 후 diff에서 요청한 줄만 바뀌었는지 다시 확인합니다.

## 금지 사항

- `???`를 임시 문자열로 소스에 넣지 않습니다.
- 기존 정상 한글을 유니코드 이스케이프 형태로 바꾸지 않습니다.
- 터미널에서 깨져 보인다는 이유만으로 문장을 새로 번역하거나 추측해서 바꾸지 않습니다.
- 한글 수정 작업 중에 관련 없는 다른 한글 문자열을 같이 건드리지 않습니다.

## 권장 방식

- 한글 수정 요청은 가능하면 “이 블록만”, “이 줄만”처럼 좁게 처리합니다.
- 문자열이 민감한 파일은 필요한 부분만 수정하고, 나머지는 그대로 둡니다.
- 새 한글 문구를 넣을 때는 UTF-8 한글 리터럴을 그대로 사용합니다.

## 체크리스트

한글 관련 수정 전후로 아래를 확인합니다.

1. 이 수정이 정말 요청 범위 안에 있는가
2. 기존 정상 한글을 불필요하게 바꾸지 않았는가
3. `???` 같은 placeholder가 남아 있지 않은가
4. 브라우저/에디터 기준으로 정상 표시되는가
5. BOM이나 불필요한 대규모 포매팅이 생기지 않았는가
