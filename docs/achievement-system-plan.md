# 업적 시스템 계획

## 목표

업적 시스템을 추가해서 플레이어가 추가 토큰을 얻고, 장기적으로 게임을 계속 플레이할 동기를 제공한다.

기본 UX:
- 로비에서 `업적` 창을 연다.
- 업적 창 안에는 여러 개의 업적이 리스트 형태로 표시된다.
- 각 업적은 다음 정보를 가진다.
  - 업적 이름
  - 업적 설명 / 조건
  - 진행도
  - 보상 토큰 수
  - 우측의 보상 수령 버튼
- 업적 조건을 달성하면 플레이어는 해당 업적의 보상 버튼을 눌러 토큰을 수령할 수 있다.
- 업적 창 상단에는 반드시 `모든 보상 획득` 버튼이 있어야 한다.

`모든 보상 획득` 버튼이 중요한 이유:
- 여러 업적이 한 번에 완료될 수 있다.
- 하나씩 받는 방식은 지루하다.
- 보상 획득 경험을 더 좋게 만든다.

## 추천 구조

업적 시스템은 아래 3개로 나눠서 생각하는 것이 가장 안전하다.

1. 업적 정의
2. 플레이어 업적 상태
3. 서버 보상 수령 검증

이 구조가 가장 안전하고 유지보수하기 쉽다.

## 중요한 설계 판단

질문:
- 업적마다 DB 값을 하나씩 전부 만들어야 하나?

답:
- 업적 정의 자체를 모두 DB row로 만들 필요는 없다.
- 하지만 플레이어별 달성 상태 / 수령 상태는 DB에 저장하는 것이 맞다.

정리하면:
- 업적 목록과 조건은 코드 또는 하나의 카탈로그 테이블에 둔다.
- 플레이어가 그 업적을 달성했는지, 보상을 받았는지는 DB에 둔다.

## 1차 추천 방식

### 업적 정의는 코드에 둔다

초기 버전에서는 업적 정의를 코드에 두는 것을 추천한다.

이유:
- 구현이 빠르다.
- git으로 변경 이력을 관리하기 쉽다.
- AI나 사람이 리뷰하기 쉽다.
- 업적은 자주 바뀌는 데이터가 아닐 가능성이 높다.

예시 필드:
- `id`
- `category`
- `title_kr`
- `title_en`
- `desc_kr`
- `desc_en`
- `reward_tokens`
- `goal`
- `progress_type`
- `hidden`
- `sort_order`

예시 `progress_type`:
- `wins_total`
- `duel_wins`
- `ai_wins`
- `ability_wins`
- `two_vs_two_wins`
- `coop_clears`
- `tokens_earned_total`
- `skins_owned`
- `tutorial_complete`
- `special_event_x`

## DB에는 무엇을 저장해야 하나

DB는 단순 UI 정보가 아니라, 플레이어의 실제 업적 상태를 저장해야 한다.

최소한 필요한 정보:
- 어떤 업적을 달성했는가
- 어떤 업적 보상을 이미 받았는가
- 현재 진행도는 얼마인가

## 왜 보상 수령 여부는 DB가 맞나

이건 네 판단이 맞다.

보상 수령 여부를 DB에서 확인해야 하는 이유:
- 클라이언트만 믿으면 조작 가능하다.
- localStorage나 프론트 상태는 안전하지 않다.
- 토큰은 게임 경제 데이터이기 때문에 서버가 최종 권한을 가져야 한다.

안전한 규칙:
- 클라이언트는 업적 UI를 보여준다.
- 서버는 수령 가능한지 검증한다.
- 서버가 토큰을 지급한다.
- 서버가 `claimed` 상태를 기록한다.

## 권장 보상 수령 흐름

### 단일 업적 보상 수령

1. 클라이언트가 업적 보상 버튼을 누른다.
2. 클라이언트가 `achievement_id`를 서버에 보낸다.
3. 서버가 아래를 검증한다.
   - 업적이 실제로 존재하는가
   - 플레이어가 해당 업적을 완료했는가
   - 이미 수령한 업적은 아닌가
4. 서버가 토큰을 지급한다.
5. 서버가 `claimed = true`로 바꾼다.
6. 서버가 새 토큰 수와 업적 상태를 반환한다.

### 모든 보상 획득

1. 클라이언트가 `모든 보상 획득` 버튼을 누른다.
2. 클라이언트가 서버에 한 번 요청한다.
3. 서버가 `완료 + 미수령` 업적을 전부 찾는다.
4. 서버가 한 번에 토큰을 지급한다.
5. 서버가 해당 업적들을 전부 `claimed = true`로 바꾼다.
6. 서버가 아래 정보를 반환한다.
   - 수령 처리된 업적 id 목록
   - 이번에 얻은 총 토큰 수
   - 갱신된 토큰 수

`모든 보상 획득`은 가능하면 서버에서 한 트랜잭션으로 처리하는 것이 좋다.

## 업적 진행도 추적 전략

모든 업적이 별도 전용 테이블을 필요로 하지는 않는다.

### A. 기존 데이터로 계산 가능한 업적

예:
- 총 승리 수
- AI 승리 수
- 대결전 승리 수
- 능력대전 승리 수
- 2v2 승리 수
- 보유 스킨 수

이런 업적은 기존 데이터에서 계산 가능할 수 있다.
예:
- `player_stats`
- `owned_skins`
- 매치 결과 기록 테이블이 있다면 그것

### B. 별도 진행도 저장이 필요한 업적

예:
- 누적 로그인 횟수
- 특정 행동 누적 횟수
- 이벤트 한정 업적
- 숨겨진 업적
- 특수 조건 업적

이런 업적은 별도 진행도 저장이 필요하다.

방법:
- `player_achievements.progress`에 직접 저장
- 또는 별도 progress 전용 테이블 추가

## 추천 1차 구조 요약

가장 현실적인 첫 버전:

- 업적 정의: 코드
- 플레이어 업적 상태: DB
- 보상 수령 검증: 서버
- `모든 보상 획득`: 서버 처리

이 구조의 장점:
- 보안적으로 안전하다.
- 구현 난이도가 적당하다.
- 나중에 확장하기 쉽다.

## 보안 규칙

아래 규칙은 꼭 지켜야 한다.

- 토큰 지급은 서버에서만 한다.
- 수령 검증은 서버에서만 한다.
- 클라이언트가 `claimed`를 직접 바꿀 수 없어야 한다.
- `모든 보상 획득`은 중복 지급이 없어야 한다.
- 가능하면 트랜잭션 안에서 처리한다.

## UI 메모

업적 카드에는 최소한 아래가 보여야 한다.
- 업적 이름
- 짧은 조건 설명
- 진행도 `x / y`
- 보상 토큰 수
- 보상 버튼

상단에는 아래가 있으면 좋다.
- 완료 개수
- 미수령 개수
- 전체 개수
- `모든 보상 획득` 버튼

나중에 추가 가능한 필터:
- 전체
- 진행 중
- 완료
- 미수령
- 수령 완료

## 추천 구현 순서

1. 업적 목록 초안 작성
2. `player_achievements` 테이블 설계
3. 서버 `claim` / `claim all` 로직 설계
4. 로비 업적 버튼 + 업적창 UI 추가
5. 진행도 계산 및 렌더링 연결
